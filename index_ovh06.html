🚀 Procédure de mise à jour

ÉTAPE 1️⃣ : LOCAL - Préparer la nouvelle image
Shell Motif Explications 
docker images
Local Récupérer REPOSITORY et TAG de ton image Affiche toutes les images disponibles 
docker ps -a
Local Permet d'afficher la liste des conteneurs exécutés / arrêtés, avec les statuts suivants :
  • Exited (0) : aucune erreur
  • Exited (1) : au moins une erreur présente
  • Exited (137) : conteneur arrêté / forcé / manque de mémoire
  • ps : vient de “process status”. Historiquement, ps est une commande Unix qui affiche l’état des processus en cours, et Docker a repris ce nom pour afficher l’état des conteneurs
  • -a : signifie 'all'
 
docker logs [CONTAINER_ID]
Local Si un des conteneurs a le statut Exited (1) ou Exited (137) - 
docker save [REPOSITORY]:[TAG] -o [REPOSITORY].tar
Local Exporter l'image en fichier .tar -o : output 
ls [REPOSITORY].tar
Local Vérifier que le .tar existe ls : liste détaillée avec taille lisible. Si absent, l'export a échoué 
scp -i $env:USERPROFILE\.ssh\[CLE] [REPOSITORY].tar ubuntu@[IPV4]:/home/ubuntu/
Local Copier le fichier .tar vers le VM scp : Secure Copy via SSH 
ssh -i $env:USERPROFILE\.ssh\[CLE] ubuntu@[IPV4]
Local Se connecter au VM en SSH Tu entres maintenant dans le shell du VM
ÉTAPE 2️⃣ : VM - Charger la nouvelle image
Shell Motif Explications 
docker load -i [REPOSITORY].tar
VM Charger la nouvelle image Docker Crée l'image v1 dans Docker. L'ancienne (v0) reste aussi disponible 
docker images | grep [IMAGE]
VM Vérifier que les 2 images coexistent Affiche 
...version0:0 et 
...version0:1 ✅
 
docker ps
VM Voir le conteneur actif (ancienne version) -
ÉTAPE 3️⃣ : VM - Arrêter l'ancienne version (⚠️ DOWNTIME ~10-30 sec)
Shell Motif Explications 
docker stop [ancien CONTAINER ID]
VM Arrêter le conteneur actif ATTENTION : À ce moment, l'app est DOWN (non accessible). Durée : ~5-10 sec. Nginx affichera "502 Bad Gateway" pour les utilisateurs
ÉTAPE 4️⃣ : VM - Lancer la nouvelle version
Shell Motif Explications 
docker run -d \
--restart unless-stopped \
-p [PORT RESEAU]:[PORT LOCAL] \
-v /home/ubuntu/enregistrements_formulaires:/app/Enregistrements \
-e APP_BASE_URL="https://[SOUS_DOMAINE].fr" \
-e COOKIE_SECURE=1 \
--name [CONTAINER NAME] \
[REPOSITORY]:[TAG]
VM Lancer le nouveau conteneur
  • -d : mode détaché (arrière-plan)
  • --restart unless-stopped : relance auto si crash
  • -p PORT RESEAU:PORT LOCAL : mappe port interne au port VM (se lit à l'envers)
  • -v /home/ubuntu/enregistrements_formulaires:/app/Enregistrements : si export des données en tant qu'admin
  • -e APP_BASE_URL="https://[SOUS_DOMAINE].fr" : si présence d'une variable d'environnement
  • -e COOKIE_SECURE=1 : si présence de cookies
  • --name [CONTAINER NAME] : crée le conteneur avec un nouveau nom

⏱️ 3-10 sec pour démarrer 
docker ps
VM Vérifier que la nouvelle version du conteneur tourne Doit afficher 
[CONTAINER NAME] avec 
STATUS: Up X seconds 
                    
✅ L'app est de retour en ligne !
ÉTAPE 5️⃣ : VÉRIFICATION - Tester la nouvelle version
Shell Motif Explications 
curl https://[SOUS-DOMAINE].[DOMAINE].fr/docs
@ Tester l'accès directement sur @ Doit retourner du HTML 
docker logs [CONTAINER NAME]
VM Si erreur 502 ou timeout Affiche les logs du conteneur pour diagnostiquer le problème
ÉTAPE 6️⃣ : OPTIONNEL - Nettoyer les anciennes versions
Shell Motif Explications 
docker ps -a | grep [IMAGE]
VM Lister tous les conteneurs avec le même nom commun pour l'image (actifs + arrêtés) - 
docker rm [ancien CONTAINER ID]
VM Supprimer le conteneur arrêté (ancien) - 
docker rmi [Ancien REPOSITORY]:[Ancien TAG]
VM Supprimer l'image Docker ancienne (optionnel) rmi = remove image. Libère l'espace dans l'instance ciblée.
⚠️ Vérifier que l'image n'est pas utilisée dans un autre sous-domaine avant de la supprimer 
df -h /
VM ⚠️ Vérifier l'espace disque libéré df -h : affiche l'utilisation disque. La colonne "Use%" doit diminuer après le nettoyage

📋 Résumé des impacts

✅ Inchangé Nginx, Certbot, Firewall UFW, certificat SSL, domaine
⚠️ Downtime ~10-30 secondes (temps pour arrêter l'ancien conteneur et en lancer un nouveau)
✅ Transparent Pour l'utilisateur, c'est comme si rien ne s'était passé. HTTPS reste actif pendant toute la mise à jour (Nginx = reverse proxy, l'accès continue).

🔄 Rollback en cas de problème (urgent)

Si la v1 a un bug critique :
  • docker stop [CONTAINER NAME] : arrête la version bugguée
  • docker start [Ancien CONTAINER NAME] : relance l'ancienne version (si pas supprimée à l'étape 6)
  • Nginx redirige immédiatement vers le port [PORT RESEAU] → ancien conteneur ✅
  • Durée : ~10 sec. Dépannage rapide !
